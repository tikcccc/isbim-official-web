# 部署文档索引

**最后更新：** 2026-03-20

## 1. 目录目的

本目录用于收敛 `isbim-official-web` 的部署、镜像构建、CCE 运行、资源清单和环境变量分层说明。

这套写法参考了 `Jarvis` 项目的 `web` 侧部署方式，但已按 isBIM 当前真实边界改造：

- 当前仓库只部署一个 `Next.js SSR` 工作负载，没有 Jarvis 那种同仓库 `cms` 容器。
- 内容系统是外部托管的 `Sanity`，不是 Strapi。
- 运行时还需要考虑 `Resend / Brevo` 邮件服务和 `Sanity webhook`，这是 Jarvis `web` 没有的差异。

## 4. 单一事实来源规则

部署相关文档要尽量贴近真实仓库状态，避免“文档写法”和“代码行为”分叉。当前按下面规则维护：

- 镜像构建行为以 `../build.sh` 和 `../Dockerfile` 为准。
- 本地开发 env 模板以 `../.env.local.example` 为准。
- 本地直接生产模式 env 模板以 `../.env.production.example` 为准。
- 镜像构建变量模板以 `../.env.build.example` 为准。
- 本地 Docker 运行时变量模板以 `../.env.runtime.example` 为准。
- 华为云 CCE 运行时配置以 `../deploy/cce/configmap.yaml` 和 `../deploy/cce/secret.example.yaml` 为准。
- `src/lib/env.ts` 是代码层读取环境变量的单一入口。
- 媒体/CDN 变量行为以 `../src/lib/media-config.ts` 和 `../next.config.ts` 为准。
- Sanity webhook 行为以 `../src/app/api/revalidate/route.ts` 为准。
- 联络表单写入 Sanity、发送邮件的运行时依赖以 `../src/app/actions/contact-form.action.ts`、`../src/lib/email/*` 为准。
- `.env.local`、`.env.production`、`.env.build.*`、`.env.runtime` 这类带实值的文件都是本机或 CI/CD 工作副本，不作为仓库标准事实来源。

## 5. 当前已知边界

- 当前仓库的部署单元只有一个：`isbim-web` 对应的 `Next.js SSR` 容器。
- `Sanity` 是外部托管服务，部署资源清单里不会出现自建 CMS Pod。
- `NEXT_PUBLIC_*` 变量必须在镜像构建前锁定，因为它们会进入 `next build` 产物。
- 运行时除了站点本身，还必须考虑 `Sanity API Token`、`Sanity webhook secret`、`Resend/Brevo API key`、邮件发件人与收件人配置。
- 当前联络表单限流仍是内存态，CCE 仍建议单副本；多副本要先接入 `Redis / 华为云 DCS`。
- 当前仓库已具备 `deploy/cce/` 最小可用清单，但 CI/CD 和真实集群发布仍需要按环境补实值。
